# Set a Validity Window on a Transaction

Every Cardano transaction can carry a time window that says "this transaction is only valid between slot X and slot Y." Outside that window, a node will reject it at phase-1 validation — even if everything else is correct.

Validity windows matter for any transaction that must land by a deadline, or must *not* land before some condition is met. Time-locked vesting, auction bids, mint cut-offs, scheduled payments — all use validity intervals as the ledger-level enforcement point.

---

## What the Window Is Made Of

A Cardano transaction body has two optional fields:

| Field | Meaning | Apollo method |
|---|---|---|
| `invalidBefore` | Transaction invalid *before* this slot (inclusive) | `SetValidityStart(slot int64)` |
| `invalidHereafter` (TTL) | Transaction invalid *after* this slot (exclusive) | `SetTtl(slot int64)` |

Either or both may be set. If both are absent, the transaction is valid in any slot (subject to the other phase-1 rules).

Slot numbers are plain `int64` — absolute slots since genesis on your target network. Preprod slots advance one per second, so you convert "30 seconds from now" into "current tip slot + 30" before building.

## Step 1: Find the Current Slot

Validity intervals are absolute slot numbers, so your program needs to know the current slot before it can pick a sensible range.

Two options:

**Option A — query Dolos via gOuroboros (matches the rest of the course):**

```go
o, _ := ouroboros.NewConnection(
    ouroboros.WithNetworkMagic(1),
    ouroboros.WithErrorChan(errChan),
    ouroboros.WithNodeToNode(false),
)
_ = o.Dial("unix", socketPath)

tip, _ := o.ChainSync().Client.GetCurrentTip()
currentSlot := int64(tip.Point.Slot)
```

**Option B — ask Blockfrost (quick, no local node needed):**

```go
// GET /epochs/latest — the response includes `slot`, the tip slot
```

For this lesson, use Option A if you already have Dolos running; otherwise the Blockfrost path is fine.

## Step 2: Set a 5-Minute Window

Open the Apollo program from 102.2. Before `.Complete()`, add the two calls:

```go
currentSlot := /* from Step 1 */ int64(tip.Point.Slot)

apollob, _, err = apollob.
    AddLoadedUTxOs(utxos...).
    PayToAddressBech32("your address here", 1_000_000).
    SetValidityStart(currentSlot).
    SetTtl(currentSlot + 300). // 5 minutes on preprod
    Complete()
```

> `Complete()` returns three values: the updated `*Apollo`, a `[]byte` (CBOR preview of the built tx body — useful for debugging, ignorable for this lesson), and an error. Discarding the middle value with `_` is the pattern you'll see throughout the course.

**What this says:** the transaction is valid starting right now (current slot), and expires 300 slots (~5 minutes) from now. If a node tries to apply this transaction before `currentSlot` or at/after `currentSlot + 300`, it rejects with `OutsideValidityInterval`.

Both methods return `*Apollo` and chain cleanly into the rest of the builder. Order doesn't matter among builder calls; what matters is that both are set **before** `.Complete()`, because `Complete()` uses them when computing the fee and the script-data hash.

## Why Both Ends Matter

Most real-world transactions use TTL only — "valid until X" — because the usual concern is "the fee estimate may stale, don't let this linger forever." But `invalidBefore` is load-bearing in a few specific cases:

- **Time-locked spending.** If a contract locks funds until slot Y, any spend attempt before slot Y is rejected by the ledger, not just the contract's on-chain logic.
- **Coordinated multi-party transactions.** If several parties must sign in sequence and each step has its own deadline, `invalidBefore` prevents a later signature from applying the earlier one "too early."
- **Minting policies that enforce a launch time.** Pair with a native-script `InvalidBefore` clause on the policy for belt-and-braces enforcement.

In all these cases, `invalidBefore` is a cheap ledger-level guardrail. Set it explicitly rather than relying on off-chain checks.

## Common Pitfalls

### Setting TTL too tight
If you set `SetTtl(currentSlot + 10)` and the mempool is even slightly congested, your tx will expire before a block producer includes it. 300 slots (~5 min) is a reasonable default for interactive use. Automated systems often use 600–1200.

### Mixing up slots and epochs
`SetTtl` takes an absolute slot, not an epoch number, not a Unix timestamp. On preprod one slot is one second; on mainnet it's the same but the slot numbering differs. Never hard-code a slot value from another network.

### Forgetting to refresh before long-running retries
If your service builds a tx, then retries for 20 minutes, your original `currentSlot + 300` TTL expires. On retry you should rebuild with a fresh tip slot.

### Ignoring the clock skew on your machine
Apollo trusts the slot number you pass. If you compute it from Unix time instead of querying the chain, and your machine's clock drifts, you'll get a TTL that doesn't match the node's view. Always derive from the chain tip.

## You'll Know You're Successful When

- Your transaction submits successfully while the window is open
- If you deliberately set `SetTtl(currentSlot - 100)`, you see an `OutsideValidityInterval` error at submission time
- You can explain the difference between `SetValidityStart` and `SetTtl` in one sentence each
