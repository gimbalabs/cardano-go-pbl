# Extract Datums and Redeemers from CBOR in Go

Every piece of data on Cardano's chain — transactions, datums, redeemers, blocks — lives as **CBOR** (Concise Binary Object Representation, RFC 8949). When your Go service sees a transaction arrive from Adder, the datums attached to its outputs and the redeemers in its witnesses are raw CBOR bytes. To interpret them, you decode those bytes into Go values.

This lesson walks through decoding two concrete things: a datum attached to an output, and a redeemer attached to a script witness. You'll use `fxamacker/cbor` (the same library Apollo uses internally) and match the Go shape to what the on-chain validator expects.

## CBOR Major Types — the 30-Second Version

CBOR splits all values into eight major types. The ones you'll see constantly on Cardano:

| Major type | Meaning | Example bytes |
|---|---|---|
| 0 | Unsigned integer | `18 7b` = 123 |
| 2 | Byte string | `48 01020304...` = 8 bytes |
| 3 | Text string | `64 74657374` = "test" |
| 4 | Array | `83 ...` = 3-element array |
| 5 | Map | `a2 ...` = 2-key map |
| 6 | Tag (semantic label wrapping another value) | `d8 7b ...` = tagged value |

Plutus data uses tags extensively. A `Constr` value (the "constructor" shape you saw in 203.1) is encoded as a CBOR tag, usually `121 + index`, wrapping an array of field values. A 28-byte verification-key hash is a byte string (major type 2).

You don't need to read CBOR bytes by hand for this lesson — the library does it. But knowing the major-type shape helps when a decode fails and you need to inspect the raw bytes with `xxd`.

## Example: Decoding a Simple Datum

Imagine the `hello_world` validator from 203.1. Its datum has one field: a 28-byte `owner` hash. On-chain, that's encoded as:

```
d8 7b          # tag 123 (= 121 + constructor index 0, but +2 for indefinite-length compatibility in some encoders)
9f             # begin indefinite-length array
58 1c          # byte string, 28 bytes
  <28 bytes of owner hash>
ff             # break (end of indefinite array)
```

(Exact tag numbers vary with encoder; Plutus uses `121 + index` as the logical tag, which CBOR encodes using different byte prefixes depending on range.)

In Go, you have a few options for decoding. The cleanest when you know the schema is to define a Go struct that mirrors the Plutus shape and let `fxamacker/cbor` do the work.

### Option A — Target struct with CBOR tags

```go
package main

import (
    "encoding/hex"
    "fmt"

    "github.com/fxamacker/cbor/v2"
)

// HelloWorldDatum matches:
//   pub type Datum { owner: VerificationKeyHash }
// Plutus constructor index 0, one byte-string field.
type HelloWorldDatum struct {
    _     struct{} `cbor:",toarray"`       // encode as CBOR array, not map
    Owner []byte                           // the 28-byte hash
}

func main() {
    datumHex := "d8799f581c0deadbeef1234567890abcdef1234567890abcdef1234567890abff"
    datumBytes, err := hex.DecodeString(datumHex)
    if err != nil {
        panic(err)
    }

    // The outer tag (d87 9f... ff) wraps an array. We strip the tag first.
    var tagged cbor.RawTag
    if err := cbor.Unmarshal(datumBytes, &tagged); err != nil {
        panic(err)
    }
    fmt.Printf("constructor tag: %d\n", tagged.Number) // 121 == index 0

    var datum HelloWorldDatum
    if err := cbor.Unmarshal(tagged.Content, &datum); err != nil {
        panic(err)
    }
    fmt.Printf("owner: %x\n", datum.Owner)
}
```

Key details:

- The `cbor:",toarray"` struct tag tells the library the value is a CBOR array (major type 4), not a map — Plutus constructors are positional, and the field order in the struct must match the blueprint field order.
- `cbor.RawTag` captures the semantic tag separately from the contained value. You read `tagged.Number` (121 for constructor index 0, 122 for index 1, etc.), then decode `tagged.Content` into the struct for that constructor.
- `[]byte` maps to CBOR byte string automatically. A text string would be `string`.

### Option B — Generic decoding when the shape is unknown

If you're inspecting unfamiliar datums, decode to `any` first and print the shape:

```go
var generic any
if err := cbor.Unmarshal(datumBytes, &generic); err != nil {
    panic(err)
}
fmt.Printf("%#v\n", generic)
```

You'll see something like:

```
cbor.Tag{Number:121, Content:[]interface{}{[]uint8{0xde, 0xad, 0xbe, 0xef...}}}
```

Useful for exploring a datum you've never seen before. Once you know the shape, switch to the typed decode from Option A.

## Example: Decoding a Redeemer

A Cardano transaction's witnesses include a redeemer for each script input or mint. Redeemers are Plutus data just like datums, so the decoding pattern is identical — the only difference is which field of which structure you're reading from.

If you're getting a redeemer from Adder, it's inside the transaction witness set. Adder's `TransactionEvent` exposes `Witnesses lcommon.TransactionWitnessSet`. Walk to the `Redeemers` field and you'll find raw CBOR per redeemer, each with a tag (Spend/Mint/Cert/Reward) and a `Data` field.

For the `hello_world` example, the redeemer is:

```rust
pub type Redeemer { msg: ByteArray }
```

Same shape as the datum: constructor index 0, one byte-string field. The Go struct is identical:

```go
type HelloWorldRedeemer struct {
    _   struct{} `cbor:",toarray"`
    Msg []byte
}
```

And `string(redeemer.Msg)` should equal `"Hello, World!"` if you're decoding a successful unlock transaction.

## Why This Matters

You need decoded datums and redeemers for three common jobs:

1. **Dispatch inside a handler.** Many contracts encode an action variant in the redeemer (constructor index 0 = deposit, index 1 = withdraw, etc.). You read the tag and branch.
2. **Indexing into a database.** You want to store "who owns this escrow UTxO" — that's reading the `owner` field out of the datum and persisting it.
3. **Building the next transaction.** To unlock a UTxO you must *exactly reconstruct* its datum. Decoding the stored one and re-encoding it with a small modification is more reliable than rebuilding from scratch.

---

## Common Pitfalls

### Wrong field order
Plutus constructors are positional — the first field in the blueprint is the first field in the array. Your Go struct must preserve that order. Renaming fields doesn't matter; reordering them does.

### Forgetting `cbor:",toarray"`
Without it, the library assumes map encoding. The decode silently produces a zero struct. Always annotate struct fields that map to Plutus `Constr`.

### Bytes vs. strings
`"Hello, World!"` on-chain is stored as bytes (`#bytes` in the blueprint), not text. Decode to `[]byte`, then call `string(...)` when displaying. Decoding directly to `string` works because fxamacker is permissive, but it hides intent.

### Expecting the wrong constructor index
If your decode succeeds but the tag number is unexpected, the on-chain data is a *different variant* than the one your struct describes. Handle other tag values explicitly, or raise an error — don't silently process the wrong variant.

### Indefinite vs. definite arrays
Plutus spec allows both. Some encoders emit indefinite-length arrays (with `9f ... ff` delimiters); others emit definite-length (`83 ...` for a 3-element array). `fxamacker/cbor` handles both transparently when decoding — no special flag needed.

## You'll Know You're Successful When

- Given a hex datum from an explorer, you can decode it to a Go struct whose field values match what you expect
- Given a redeemer, you can read its constructor tag and dispatch on it
- If you feed in malformed bytes, the library returns an error you can surface, not a silently-wrong struct
