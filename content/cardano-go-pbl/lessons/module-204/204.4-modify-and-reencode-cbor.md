# Modify and Re-encode CBOR Data in Go

In 204.1 you decoded CBOR values off the chain into Go structs. The natural next step is to *modify* those values — increment a counter in a datum, replace an owner, append to a list — and re-encode them for use in the next transaction. Done correctly, this gives you a round-trip: decode → modify → encode, with the encoded bytes ready to hand back to Apollo.

Done carelessly, the round-trip loses information. A decoded-and-re-encoded value must produce the **same CBOR bytes** as the original if no field changed, or you break the on-chain contract. This lesson shows you how to do that reliably.

---

## Why Round-Tripping Is Easy to Get Wrong

CBOR has multiple valid encodings for the same logical value. For example:

- An integer like `5` can be encoded as `05` (1 byte) or `18 05` (2 bytes). Both decode to `5`, but the bytes differ.
- An array `[1, 2, 3]` can be encoded as definite-length (`83 01 02 03`) or indefinite-length (`9f 01 02 03 ff`). Same logical value, different bytes.
- A map with keys `{"a": 1, "b": 2}` can be sorted or unsorted. Same value, different bytes.

Cardano's ledger uses **canonical CBOR** — a specific set of encoding rules that eliminates this ambiguity. A datum hash depends on the exact bytes; if you re-encode in a non-canonical way, the hash changes and the on-chain reference breaks.

Fortunately, `fxamacker/cbor/v2` ships with canonical encoding modes that match Cardano's rules.

## The Canonical Encoding Setup

Before encoding anything destined for a Cardano transaction, build an `EncMode` configured for canonical CBOR:

```go
import "github.com/fxamacker/cbor/v2"

var cardanoCanonical cbor.EncMode

func init() {
    opts := cbor.CanonicalEncOptions()
    // Optional: tune individual fields. Apollo's internal config is close to defaults.
    em, err := opts.EncMode()
    if err != nil {
        panic(err)
    }
    cardanoCanonical = em
}
```

`CanonicalEncOptions()` gives you:

- Shortest-form integer encoding
- Definite-length arrays and maps
- Sorted map keys (length-first then lexicographic)

These match the CDDL `.canonical` constraint that Cardano's ledger expects. Use `cardanoCanonical.Marshal(value)` for every on-chain encode.

> **Why not just `cbor.Marshal`?** The package-level `cbor.Marshal` uses default encoding options which are *not* canonical. Re-encoded datums will fail hash checks. Always go through your configured `EncMode`.

## The Round-Trip Pattern

A typical decode → modify → encode cycle:

```go
// 1. Decode into a typed Go struct (from 204.1)
type EscrowDatum struct {
    _        struct{} `cbor:",toarray"`
    Owner    []byte
    Deadline uint64
    Amount   uint64
}

var original EscrowDatum
var tagged cbor.RawTag
cbor.Unmarshal(originalBytes, &tagged)
cbor.Unmarshal(tagged.Content, &original)

// 2. Modify
modified := original
modified.Amount = original.Amount + 1_000_000  // add 1 ADA
modified.Deadline = newDeadlineSlot

// 3. Re-encode with canonical mode
innerBytes, err := cardanoCanonical.Marshal(modified)
if err != nil {
    return nil, err
}

// 4. Re-wrap in the constructor tag
wrapped := cbor.Tag{Number: tagged.Number, Content: cbor.RawMessage(innerBytes)}
outputBytes, err := cardanoCanonical.Marshal(wrapped)
```

Three points to notice:

- **Modify a copy.** `modified := original` makes a shallow copy. For a struct with only value types (bytes/ints/strings) that's safe. If your struct contains nested slices or pointers, deep-copy to avoid mutating the original.
- **Keep the constructor tag unchanged.** Unless you're deliberately changing the variant, the `Number` in the re-wrapped tag must match the decoded tag. Changing constructor index means "this is now a different variant," which is almost never what you want.
- **Use `cbor.RawMessage`, not `[]byte`, when re-wrapping.** `RawMessage` tells the encoder "this is already-encoded CBOR, splice it in as-is." A `[]byte` would be re-encoded *as a byte string*, which double-wraps and changes the meaning.

## Verifying the Round-Trip on an Unchanged Value

Before you trust the round-trip for modified values, prove it works for unchanged ones:

```go
unchanged := original

innerBytes, _ := cardanoCanonical.Marshal(unchanged)
wrapped := cbor.Tag{Number: tagged.Number, Content: cbor.RawMessage(innerBytes)}
roundTripped, _ := cardanoCanonical.Marshal(wrapped)

if bytes.Equal(roundTripped, originalBytes) {
    fmt.Println("canonical round-trip OK")
} else {
    fmt.Printf("bytes differ:\n  in:  %x\n  out: %x\n", originalBytes, roundTripped)
}
```

If the bytes differ, the original wasn't encoded canonically — common for datums produced by tooling that's less strict than Apollo. Two options:

1. Keep the original bytes alongside the struct, use the original for hash references, use the modified bytes for the new datum.
2. Accept that the hash of your re-encoded version will differ and update any on-chain references accordingly.

Most of the time you're creating a *new* datum at a *new* output, so the new hash is the one that matters.

---

## Hashing a Datum

A datum hash is `blake2b-256` of the datum's CBOR bytes — not of the decoded struct. Always hash the exact bytes you'll put on-chain:

```go
import "golang.org/x/crypto/blake2b"

h := blake2b.Sum256(outputBytes)
fmt.Printf("datum hash: %x\n", h)
```

This hash is what you reference in a transaction output's `datum_hash` field (for CIP-32 hash-style) or what you compare against a stored reference. If you hash the decoded struct via `fmt.Sprintf`, you'll get a meaningless hash.

## A Realistic Example: Incrementing a Counter

Consider a contract that stores a counter in its datum:

```rust
pub type CounterDatum { count: Int, owner: VerificationKeyHash }
```

The off-chain flow for incrementing:

1. Read the current UTxO and its datum bytes
2. Decode the datum to `CounterDatum{ Count: 42, Owner: [...]}`
3. Create `modified = CounterDatum{ Count: 43, Owner: original.Owner }`
4. Re-encode canonically, compute new hash
5. Build a transaction that consumes the original UTxO and produces a new output with the new datum

The Apollo builder handles step 5 with `AttachDatum(&newDatumPlutusData)` or `PayToContract(address, amount, datum)` — but it's *your* responsibility to ensure the datum you pass in is the canonically-encoded, correctly-modified value.

## Common Pitfalls

### Using default `cbor.Marshal` for on-chain bytes
Produces non-canonical CBOR. Silent bug until you compute a hash and it doesn't match.

### Mutating the decoded struct in place
Fine if you know you won't need the original. Dangerous in async handlers where the decoded value might be read by another goroutine. Copy first.

### Double-wrapping a RawMessage
Encoding `cbor.Tag{Content: innerBytes}` where `innerBytes` is `[]byte` wraps the bytes as a byte string inside the tag. The decoder then sees a tag wrapping a byte string wrapping the real content — wrong. Use `cbor.RawMessage(innerBytes)`.

### Assuming encoder matches producer
If a datum was produced by a non-Go tool (cardano-cli, Lucid, mesh), its encoding may differ slightly from fxamacker's canonical output. Always verify round-trip equality on unchanged values before relying on byte-identity.

### Forgetting that constructor tag >= 127 uses different CBOR encoding
Plutus constructor indices above 6 use tag `121 + index` up to 127, and a different scheme for 7+ (wrapping in tag 102 with an index inside). The library handles this automatically when encoding Go values — but if you're constructing `cbor.Tag` manually, know that the tag numbers aren't linearly `121 + index` forever.

## You'll Know You're Successful When

- Round-tripping an unchanged datum produces byte-identical output
- Modifying a single field and re-encoding produces a datum that Apollo accepts in a transaction
- The hash of your re-encoded datum matches what a block explorer shows after submission
- You can explain in one sentence why `cbor.CanonicalEncOptions` matters
