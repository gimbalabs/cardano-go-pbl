# SLT 203.6: I Can Build a Transaction That Mints or Burns Tokens Using a Validator Script

In 203.2 you minted tokens with a native script, policy controlled by key signatures. Now you step up to a **Plutus validator**, where the minting policy is arbitrary on-chain logic. The key difference: the transaction must supply a **redeemer** that the script evaluates before allowing the mint.

This lesson uses the `hello_world` validator from 203.1. Its `mint` handler checks that the redeemer's `msg` field equals `"HelloMintRedeemer"`.

## Background: Minting with a Plutus Validator

### The mint handler vs the spend handler

In 203.5 you unlocked funds by satisfying the `spend` handler. Minting is governed by a different handler `mint` and it works differently in one critical way: **there is no datum**.

The `mint` handler has no UTxO to protect and no datum to read. It receives only:

| Argument | Source | What it contains |
|----------|--------|-----------------|
| `redeemer` | Your transaction's witness set, paired to this policy | The `PlutusData` you construct — here `{msg: "HelloMintRedeemer"}` |
| `_policy_id` | Supplied by the node from the transaction's minting map | The policy ID (script hash) being evaluated |
| `_self` (Transaction) | Built by the node from your full transaction body | All inputs, outputs, signatories, minting map, etc. |

This makes minting simpler than spending: no UTxO to fetch, no datum to construct, no inline datum storage. The Plutus VM runs the `mint` handler, checks the redeemer, and returns `True` or `False`. If `True`, the minted tokens are allowed. If `False`, the transaction fails at Phase 2.

### How the node pairs the MINT redeemer to the right policy

In 203.5 the redeemer was paired to a spending input by `Tag: Redeemer.SPEND` and input index position. For minting, the node pairs redeemers to policies using `Tag: Redeemer.MINT` and the policy ID's position in the transaction's sorted minting map. If your transaction mints under two different policies, each policy gets its own redeemer entry, indexed by position. `MintAssetsWithRedeemer` handles this pairing automatically when you pass the `mintUnit` (which carries the policy ID) alongside the redeemer.

### The policy ID comes directly from plutus.json

In 203.2 you derived the policy ID yourself by calling `script.Hash()` on a native script constructed from your key. There was no pre-existing policy ID — you computed it.

For a Plutus validator, the policy ID is the BLAKE2b-224 hash of the compiled script bytes. Aiken computes this and writes it directly into the `hash` field of `plutus.json` when you run `aiken build`. You do not derive it in Go — you read it:

```go
mintScript, policyId, err := loadScript("plutus.json", "lesson203_1.hello_world.mint")
```

The `policyId` string returned is exactly the value every token minted under this script will carry for its entire lifetime on Cardano. Every `HelloToken` minted by this script has `policyId` as part of its on-chain identity. Change one byte in the validator source, rebuild, and you get a completely different policy ID that cannot touch previously minted tokens.

### Why there is no `AddRequiredSignerFromBech32` here

In 203.5 the `spend` handler explicitly checked `list.has(self.extra_signatories, owner)` requiring the datum owner to appear in `required_signers`. That check is in the Aiken source, not in Apollo.

The `hello_world` `mint` handler checks only one thing: `redeemer.msg == "HelloMintRedeemer"`. It does not check signatories at all. So `AddRequiredSignerFromBech32` is not needed anyone who supplies the correct redeemer bytes can mint under this policy.

## The Contract

The hello_world `mint` handler from 203.1:

```aiken
mint(
  redeemer: Redeemer,
  _policy_id: PolicyId,
  _self: Transaction,
) {
  redeemer.msg == "HelloMintRedeemer"
}
```

From the blueprint, `Redeemer` is constructor 0, one `#bytes` field → tag 121. The `hash` field of `"lesson203_1.hello_world.mint"` in `plutus.json` is the policy ID.

## Complete Minting Example

```go
package main

import (
	"encoding/hex"
	"encoding/json"
	"fmt"
	"os"

	"github.com/Salvionied/apollo"
	"github.com/Salvionied/apollo/constants"
	"github.com/Salvionied/apollo/serialization/PlutusData"
	"github.com/Salvionied/apollo/serialization/Redeemer"
	"github.com/Salvionied/apollo/txBuilding/Backend/BlockFrostChainContext"
)

const (
	BLOCKFROST_KEY = "preprodYOUR_KEY_HERE"
	MNEMONIC       = "word1 word2 word3 word4 word5 word6 word7 word8 word9 word10 word11 word12"
)

type Blueprint struct {
	Validators []struct {
		Title        string `json:"title"`
		CompiledCode string `json:"compiledCode"`
		Hash         string `json:"hash"`
	} `json:"validators"`
}

// loadScript decodes a validator's compiledCode from plutus.json.
// compiledCode is hex-encoded CBOR — hex.DecodeString gives the raw bytes Apollo needs.
// Returns the script and its hash (the policy ID for mint validators).
func loadScript(path, title string) (PlutusData.PlutusV3Script, string, error) {
	data, err := os.ReadFile(path)
	if err != nil {
		return nil, "", err
	}
	var bp Blueprint
	if err := json.Unmarshal(data, &bp); err != nil {
		return nil, "", err
	}
	for _, v := range bp.Validators {
		if v.Title == title {
			code, err := hex.DecodeString(v.CompiledCode)
			if err != nil {
				return nil, "", err
			}
			return PlutusData.PlutusV3Script(code), v.Hash, nil
		}
	}
	return nil, "", fmt.Errorf("validator %q not found", title)
}

// buildMintRedeemer returns the hello_world mint redeemer.
// Blueprint: constructor 0, fields: [msg: #bytes] → tag 121
// MintAssetsWithRedeemer takes Redeemer.Redeemer with Tag: Redeemer.MINT.
func buildMintRedeemer() Redeemer.Redeemer {
	return Redeemer.Redeemer{
		Tag: Redeemer.MINT,
		Data: PlutusData.PlutusData{
			PlutusDataType: PlutusData.PlutusArray,
			TagNr:          121,
			Value: PlutusData.PlutusIndefArray{
				PlutusData.PlutusData{
					PlutusDataType: PlutusData.PlutusBytes,
					Value:          []byte("HelloMintRedeemer"),
				},
			},
		},
		ExUnits: Redeemer.ExecutionUnits{Mem: 0, Steps: 0},
	}
}

func main() {
	bfc, err := BlockFrostChainContext.NewBlockfrostChainContext(
		constants.BLOCKFROST_BASE_URL_PREPROD,
		int(constants.PREPROD),
		BLOCKFROST_KEY,
	)
	if err != nil {
		panic(err)
	}

	apollob := apollo.New(&bfc)
	apollob, err = apollob.SetWalletFromMnemonic(MNEMONIC, constants.PREPROD)
	if err != nil {
		panic(err)
	}
	apollob, err = apollob.SetWalletAsChangeAddress()
	if err != nil {
		panic(err)
	}

	// hash field from plutus.json is the policy ID directly — no derivation needed.
	// compiledCode → hex.DecodeString → PlutusV3Script(bytes)
	mintScript, policyId, err := loadScript("plutus.json", "lesson203_1.hello_world.mint")
	if err != nil {
		panic(err)
	}
	fmt.Println("Policy ID:", policyId)

	utxos, err := bfc.Utxos(*apollob.GetWallet().GetAddress())
	if err != nil {
		panic(err)
	}

	redeemer := buildMintRedeemer()
	mintUnit := apollo.NewUnit(policyId, "HelloToken", 1_000)

	// MintAssetsWithRedeemer automatically handles ExUnit estimation.
	// AttachV3Script adds the compiled script bytes to the transaction witness set.
	apollob, _, err = apollob.
		AddLoadedUTxOs(utxos...).
		AttachV3Script(mintScript).
		MintAssetsWithRedeemer(mintUnit, redeemer).
		PayToAddressBech32(apollob.GetWallet().GetAddress().String(), 2_000_000, mintUnit).
		Complete()
	if err != nil {
		panic(err)
	}

	apollob = apollob.Sign()
	txId, err := apollob.Submit()
	if err != nil {
		panic(err)
	}

	fmt.Println("Tx hash:", hex.EncodeToString(txId.Payload))
}
```

Run it:

```bash
go run .
```

Check [preprod.cardanoscan.io](https://preprod.cardanoscan.io). The policy ID should match the printed hash. Quantity: 1,000 `HelloToken`.

## What Each Step Does

`loadScript`, `AttachV3Script`, `Complete()`, and the ExUnit estimation flow were explained in detail in 203.5. This section covers only the steps that are new or behave differently for minting.

---

### `buildMintRedeemer()` — `MINT` tag, not `SPEND`

```go
Redeemer.Redeemer{
    Tag: Redeemer.MINT,
    ...
}
```

The only structural change from the spend redeemer in 203.5 is the `Tag`. `Redeemer.MINT` tells the node this redeemer belongs to a minting policy, not a spending input.

The `Data` field is built the same way from the blueprint: constructor 0, one `#bytes` field, tag 121 — the same `PlutusData` structure covered in 203.4.

---

### `MintAssetsWithRedeemer(mintUnit, redeemer)` — the core difference from 203.2

```go
MintAssetsWithRedeemer(mintUnit, redeemer)
```

In 203.2 you used `MintAssetsWithNativeScript(mintUnit, script)`. The native script call embedded the script as a witness and the node evaluated it through declarative rule checking — no Plutus VM, no ExUnits, no collateral required.

`MintAssetsWithRedeemer` does something fundamentally different:

1. Registers the mint in the transaction body under the policy ID embedded in `mintUnit`
2. Pairs the redeemer with that policy ID (indexed by its position in the sorted minting map)
3. Signals to `Complete()` that ExUnit estimation is required — the Plutus VM will run

When `Complete()` fires, it calls Blockfrost's off-chain evaluate endpoint to simulate the `mint` handler execution and fill in the estimated `ExUnits`. Those ExUnits become the budget cap the node enforces during Phase 2 on-chain, and they feed into the script fee component of the total transaction fee.

## Burning Tokens

Use a negative quantity with the same redeemer and script:

```go
burnUnit := apollo.NewUnit(policyId, "HelloToken", -500)

apollob, _, err = apollob.
    AddLoadedUTxOs(utxos...).
    AttachV3Script(mintScript).
    MintAssetsWithRedeemer(burnUnit, redeemer).
    Complete()
```

The tokens to burn must be present in a UTxO in `AddLoadedUTxOs`.

## Native Script vs Validator Script

| | Native Script (203.2) | Validator Script (203.6) |
|---|---|---|
| Policy enforced by | Key signatures / time locks | Arbitrary on-chain Aiken logic |
| Redeemer required | No | Yes — `PlutusData` |
| Script attached via | Built into `MintAssetsWithNativeScript` | `AttachV3Script` |
| Apollo mint method | `MintAssetsWithNativeScript` | `MintAssetsWithRedeemer` |
| Policy ID source | `script.Hash()` derived from key | `hash` field in `plutus.json` |

## Common Errors

**`Script not found in witness set`** — `AttachV3Script` must be called before `Complete()`. Without it, the node's Phase 1 check fails because the policy ID in the minting map has no matching script bytes in the witness set, and the Plutus VM never runs.

**`Redeemer data mismatch`** — the `msg` bytes must exactly match what the validator checks. `"HelloMintRedeemer"` and `"HelloMintRedeemer "` (trailing space) are different byte sequences.

**`Wrong script version`** — if `plutus.json` says `"plutusVersion": "v3"` use `PlutusV3Script` / `AttachV3Script`. For `v2` use `PlutusV2Script` / `AttachV2Script`.

**`tokens not sent to output`** — minted tokens must appear in a transaction output. Include `mintUnit` as an extra argument to `PayToAddressBech32`.
