# SLT 203.5: I Can Build a Transaction That Unlocks Funds Held by a Smart Contract

Sending funds to a script address is straightforward, you did it in 203.3. Getting them back requires a transaction that satisfies the validator's conditions: correct redeemer, correct signers, and the script attached so the node's Plutus VM can execute it.

This lesson covers the unlock step using the UTxO you locked in 203.3 and the redeemer patterns you learned in 203.4.

## Background: eUTxO Spending

This is one of the most important concepts in Cardano smart contracts. Understanding exactly what happens here will prevent the majority of script failures you will ever encounter.

### Why a script UTxO is different from a wallet UTxO

When you locked 5 ADA in 203.3, those funds moved from your wallet address to a **script address**. Script addresses have no private key no one can sign for them. The only way to consume a UTxO sitting at a script address is to provide a transaction that the **script itself** approves.

The Cardano node enforces this through **two-phase validation**:

- **Phase 1** — the node checks basic structural validity: inputs exist, fees are covered, signatures are present, and every script input has a redeemer paired to it. If Phase 1 fails, the transaction is discarded immediately and no fees are charged.
- **Phase 2** — the node's built-in **Plutus VM** executes each script in the transaction. The script receives datum, redeemer, and ScriptContext and must return `True`. If Phase 2 fails, the transaction is rejected and your **collateral** UTxO is taken to compensate the node for its computation.

### The redeemer is what triggers script execution

The redeemer does more than carry data. Its presence in the transaction is the **signal to the node that a script must be evaluated**. Each redeemer is bound to a specific input via its `Tag` (`SPEND`) and position in the sorted input list this pairing tells the Plutus VM: "when processing this input, run the script with this data."

Without a redeemer paired to a script input, the Phase 1 check fails immediately — the node will not even attempt to run the script.

Think of it this way: the **UTxO at the script address is the lock**, the **script is the mechanism inside the lock**, the **datum is the combination set when the lock was closed**, and the **redeemer is the key you insert to trigger the mechanism**. The Plutus VM checks whether your key fits.

### What the Plutus VM runs

When the node's Plutus VM executes the script during Phase 2, it calls the `spend` handler with three arguments it assembles from your transaction:

| Argument | Source | What it contains |
|----------|--------|-----------------|
| `datum` | Read from the UTxO's inline datum (set in 203.3) | `{owner: <your-pkh>}` — the 28-byte payment key hash you set |
| `redeemer` | Your transaction's witness set, paired to this input | `{msg: "HelloSpendRedeemer"}` — the bytes you provide now |
| `ScriptContext` | Built by the node from your full transaction body | All inputs, outputs, signatories, validity range, etc. |

The script computes: `redeemer.msg == "HelloSpendRedeemer" && list.has(extra_signatories, owner)`. If both conditions are `True`, the UTxO is consumed and the transaction is applied to the chain. If either is `False`, Phase 2 fails — the entire transaction is rejected, no partial execution happens, and your collateral is taken.

### ExUnits: simulation before submission, budget cap on-chain

Before you submit the transaction, `Complete()` calls Blockfrost's evaluate endpoint which runs the Plutus VM **off-chain** against your transaction to *estimate* how many `ExUnits` (memory units and CPU steps) the script will consume. Those estimates are written into the `ExUnits` field of your redeemer.

When the node receives the submitted transaction, it runs the Plutus VM again on-chain during Phase 2. It checks that the actual ExUnits consumed **do not exceed** the declared ExUnits — the declared value is an execution budget cap. The declared ExUnits also determine the script fee component that is added to your base transaction fee:

```
script_fee = mem_price × ExUnits.mem + step_price × ExUnits.steps
```

This is why you set `ExUnits{Mem: 0, Steps: 0}` in your redeemer struct — you are not guessing; you are letting `Complete()` fill in accurate estimates via simulation before the transaction is finalised.

### Four things Apollo must assemble

Because Cardano's Phase 1 and Phase 2 validation rules are strict about what a script-spending transaction must contain, Apollo needs you to provide four things explicitly:

1. **The UTxO being spent** — `CollectFrom` identifies which locked UTxO this transaction is consuming and binds the redeemer to it
2. **The redeemer** — the `Redeemer.Redeemer` struct carrying `{msg: "HelloSpendRedeemer"}` with `Tag: Redeemer.SPEND`
3. **The script bytes** — `AttachV3Script` places the compiled script in the witness set so the node's Plutus VM can execute it
4. **The required signer** — `AddRequiredSignerFromBech32` adds your payment key hash to `extra_signatories` so the script's signature check passes

All four are mandatory. Missing any one of them produces a different error, all covered below.

## The Contract (from 203.1)

```aiken
validator hello_world {
  spend(datum: Option<Datum>, redeemer: Redeemer, _own_ref: OutputReference, self: Transaction) {
    expect Some(Datum { owner }) = datum
    let must_say_hello = redeemer.msg == "HelloSpendRedeemer"
    let must_be_signed = list.has(self.extra_signatories, owner)
    must_say_hello? && must_be_signed?
  }
}
```

Blueprint types:
```
Datum:    constructor 0, fields: [owner: #bytes]   → tag 121
Redeemer: constructor 0, fields: [msg: #bytes]     → tag 121
```

## Unlock Transaction

Replace `YOUR_LOCK_TX_HASH` with the hash printed by the 203.3 lock transaction.

```go
package main

import (
	"encoding/hex"
	"encoding/json"
	"fmt"
	"os"

	"github.com/Salvionied/apollo"
	"github.com/Salvionied/apollo/constants"
	"github.com/Salvionied/apollo/serialization/PlutusData"
	"github.com/Salvionied/apollo/serialization/Redeemer"
	"github.com/Salvionied/apollo/txBuilding/Backend/BlockFrostChainContext"
)

const (
	BLOCKFROST_KEY = "preprodYOUR_KEY_HERE"
	MNEMONIC       = "word1 word2 word3 word4 word5 word6 word7 word8 word9 word10 word11 word12"
	LOCK_TX_HASH   = "YOUR_LOCK_TX_HASH"
	LOCK_TX_INDEX  = 0
)

type Blueprint struct {
	Validators []struct {
		Title        string `json:"title"`
		CompiledCode string `json:"compiledCode"`
		Hash         string `json:"hash"`
	} `json:"validators"`
}

// loadScript decodes the compiledCode from plutus.json and returns it as a PlutusV3Script.
func loadScript(path, title string) (PlutusData.PlutusV3Script, error) {
	data, err := os.ReadFile(path)
	if err != nil {
		return nil, err
	}
	var bp Blueprint
	if err := json.Unmarshal(data, &bp); err != nil {
		return nil, err
	}
	for _, v := range bp.Validators {
		if v.Title == title {
			code, err := hex.DecodeString(v.CompiledCode)
			if err != nil {
				return nil, err
			}
			return PlutusData.PlutusV3Script(code), nil
		}
	}
	return nil, fmt.Errorf("validator %q not found", title)
}

// buildSpendRedeemer wraps the hello_world Redeemer PlutusData in Redeemer.Redeemer.
// Blueprint: constructor 0, fields: [msg: #bytes]. msg must be "HelloSpendRedeemer".
func buildSpendRedeemer() Redeemer.Redeemer {
	return Redeemer.Redeemer{
		Tag: Redeemer.SPEND,
		Data: PlutusData.PlutusData{
			PlutusDataType: PlutusData.PlutusArray,
			TagNr:          121,
			Value: PlutusData.PlutusIndefArray{
				PlutusData.PlutusData{
					PlutusDataType: PlutusData.PlutusBytes,
					Value:          []byte("HelloSpendRedeemer"),
				},
			},
		},
		ExUnits: Redeemer.ExecutionUnits{Mem: 0, Steps: 0},
	}
}

func main() {
	bfc, err := BlockFrostChainContext.NewBlockfrostChainContext(
		constants.BLOCKFROST_BASE_URL_PREPROD,
		int(constants.PREPROD),
		BLOCKFROST_KEY,
	)
	if err != nil {
		panic(err)
	}

	apollob := apollo.New(&bfc)
	apollob, err = apollob.SetWalletFromMnemonic(MNEMONIC, constants.PREPROD)
	if err != nil {
		panic(err)
	}
	apollob, err = apollob.SetWalletAsChangeAddress()
	if err != nil {
		panic(err)
	}

	spendScript, err := loadScript("plutus.json", "lesson203_1.hello_world.spend")
	if err != nil {
		panic(err)
	}

	// UtxoFromRef fetches the locked UTxO from the chain via Blockfrost.
	// The UTxO must be confirmed on-chain before this call — wait ~20s after locking.
	scriptUtxo, err := apollob.UtxoFromRef(LOCK_TX_HASH, LOCK_TX_INDEX)
	if err != nil {
		panic(err)
	}

	walletUtxos, err := bfc.Utxos(*apollob.GetWallet().GetAddress())
	if err != nil {
		panic(err)
	}

	redeemer := buildSpendRedeemer()

	// AddRequiredSignerFromBech32(address, addPaymentPart, addStakingPart)
	// The validator checks extra_signatories contains datum.owner.
	// AddRequiredSignerFromBech32 adds the payment key hash to that list.
	apollob, _, err = apollob.
		AddLoadedUTxOs(walletUtxos...).
		CollectFrom(*scriptUtxo, redeemer).
		AddRequiredSignerFromBech32(apollob.GetWallet().GetAddress().String(), true, false).
		AttachV3Script(spendScript).
		PayToAddressBech32(apollob.GetWallet().GetAddress().String(), 4_500_000).
		Complete()
	if err != nil {
		panic(err)
	}

	apollob = apollob.Sign()
	txId, err := apollob.Submit()
	if err != nil {
		panic(err)
	}

	fmt.Println("Unlock tx hash:", hex.EncodeToString(txId.Payload))
}
```

## What Each Step Does

### 1. `loadScript("plutus.json", "lesson203_1.hello_world.spend")`

```go
spendScript, err := loadScript("plutus.json", "lesson203_1.hello_world.spend")
```

This reads the `compiledCode` field from `plutus.json` and converts it:

```
hex string in plutus.json
  → hex.DecodeString → raw CBOR bytes
  → PlutusData.PlutusV3Script(bytes)
```

The resulting `PlutusV3Script` is just a `[]byte` with a type alias the raw compiled script. You must supply the full bytes in the transaction witness set because **the node will not look up scripts from storage** why cardano smart contract is stateless. During Phase 1 validation, the node hashes the bytes you provide and checks that the hash matches the script hash embedded in the UTxO's address. If you provide wrong bytes, or no bytes, the transaction fails at Phase 1 before the Plutus VM is ever invoked.

---

### 2. `UtxoFromRef(LOCK_TX_HASH, LOCK_TX_INDEX)`

```go
scriptUtxo, err := apollob.UtxoFromRef(LOCK_TX_HASH, LOCK_TX_INDEX)
```

This fetches the specific UTxO from Blockfrost using the transaction hash and output index from your 203.3 lock transaction. The returned `scriptUtxo` contains the UTxO's full state: the ADA value, the inline datum bytes, and the script address.
if utxo is already spent it returns nothing
---

### 3. `buildSpendRedeemer()` — constructing the key

```go
func buildSpendRedeemer() Redeemer.Redeemer {
    return Redeemer.Redeemer{
        Tag: Redeemer.SPEND,
        Data: PlutusData.PlutusData{ ... "HelloSpendRedeemer" ... },
        ExUnits: Redeemer.ExecutionUnits{Mem: 0, Steps: 0},
    }
}
```

Three fields matter here:

| Field | Value | Why |
|-------|-------|-----|
| `Tag` | `Redeemer.SPEND` | Tells the node this redeemer is for a spending input, not a minting policy. The node uses `Tag` + input index to pair the redeemer with the right script before handing it to the Plutus VM. |
| `Data` | `PlutusData` encoding `{msg: "HelloSpendRedeemer"}` | The actual argument passed to the script's `spend` handler. The script checks `redeemer.msg == "HelloSpendRedeemer"` — the bytes must match exactly. |
| `ExUnits` | `{Mem: 0, Steps: 0}` | Budget declared as zero. Apollo's `Complete()` calls Blockfrost to simulate the script and fills in the real values. You never need to compute ExUnits manually. |

---

### 4. `CollectFrom(*scriptUtxo, redeemer)` — the most important call

```go
CollectFrom(*scriptUtxo, redeemer)
```

This single call does two things at once:

1. **Registers the script UTxO as a transaction input** — the locked UTxO is added to the input set. When `Complete()` runs, this input's ADA is available to cover fees and fund the output.

2. **Binds the redeemer to this specific input** —  It is paired to this exact input by index position in the sorted input list. The node reads this pairing and knows: "when the Plutus VM executes the script for this input, pass this redeemer." Without this pairing, Phase 1 validation sees a script input with no redeemer and rejects the transaction before the Plutus VM is ever invoked.

This is why you cannot add the redeemer separately from the input. `CollectFrom` enforces the binding that Cardano's validation rules require.

---

### 5. `AddRequiredSignerFromBech32(address, true, false)` — two separate things

```go
AddRequiredSignerFromBech32(apollob.GetWallet().GetAddress().String(), true, false)
```

Signing the transaction with `apollob.Sign()` later adds a VKey witness — proof that you possess the private key. But the `hello_world` validator does not check the VKey witness. It checks `self.extra_signatories`, which is a **separate field in the transaction body** called `required_signers`.

`AddRequiredSignerFromBech32` adds your 28-byte payment key hash to `required_signers` in the transaction body. The node then exposes that field to the Plutus VM as `extra_signatories` inside the `ScriptContext`. The script calls `list.has(self.extra_signatories, owner)`, if your key hash is not in that list, the Plutus VM returns `False` and Phase 2 fails, even though your VKey signature is perfectly valid.

Both are required for this transaction:
- `AddRequiredSignerFromBech32` → puts the key hash in `required_signers` (read by the Plutus VM via `ScriptContext`)
- `apollob.Sign()` → produces the VKey witness (read by the node during Phase 1 to verify you actually own the key)

---

### 6. `AttachV3Script(spendScript)` — placing the script in the witness set

```go
AttachV3Script(spendScript)
```

You must call this even though the script's hash is already embedded in the UTxO address. Cardano stores only hashes and addresses on-chain, not full script bytecode. Your spending transaction must supply the bytecode every time you want the Plutus VM to execute it. (Reference scripts introduced in Vasil are an exception they allow a previously-stored script UTxO to be referenced instead of re-attaching the bytes, but that is a later topic.)

---

### 7. `PayToAddressBech32(address, 4_500_000)` — routing the unlocked funds

```go
PayToAddressBech32(apollob.GetWallet().GetAddress().String(), 4_500_000)
```

Once the script approves the transaction, the locked UTxO's ADA becomes a spendable input. This output sends 4.5 ADA back to your wallet. The remainder (the locked 5 ADA minus 4.5 ADA minus transaction fees) returns as change automatically. You do not need to account for fees explicitly `Complete()` handles the coin selection and change calculation.

---

### 8. `Complete()` — simulation, fee estimation, and final assembly

```go
Complete()
```

This is where Apollo does the heavy lifting:

1. Calls Blockfrost's evaluate endpoint to **simulate script execution off-chain** — Blockfrost runs the Plutus VM against your transaction body and returns *estimated* `ExUnits` (memory and CPU steps)
2. Writes those estimated `ExUnits` into the redeemer (replacing the zeros you set) — these become the declared budget the node will enforce on-chain
3. Runs **coin selection** to pick wallet UTxOs that cover the fee
4. Calculates the **minimum fee**: base fee (`a × tx_size + b`) plus the script fee (`mem_price × mem + step_price × steps`)
5. Assembles the final balanced transaction body

If the off-chain simulation fails during `Complete()` — wrong redeemer bytes, missing required signer, wrong datum structure — you will see the error here, before the transaction is ever submitted. This is intentional: Cardano's determinism means the simulation result is the same as what the node will see on-chain, so catching failures here saves you from losing collateral.

## Common Errors

**`Missing required signer`** — the validator calls `list.has(self.extra_signatories, owner)`. Without `AddRequiredSignerFromBech32`, the key hash is absent from `extra_signatories` even though the transaction is signed. Both are required.

**`UTxO not found`** — the lock transaction must confirm on-chain before you can spend it. Wait ~20 seconds and check the explorer.

**`Wrong redeemer`** — the message must be exactly `[]byte("HelloSpendRedeemer")`. Any byte difference fails the validator.

**`Script execution failed`** — verify the datum `owner` matches your wallet's payment key hash. If you locked with a different wallet, you cannot satisfy the signature check.
