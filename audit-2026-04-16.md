## (a) Wrong function names / endpoints / API shapes

### 🚨 Module 201 + 202 — Adder event-type strings (critical)

All Adder lessons (**201.2, 201.3, 201.4, 201.6, 202.3, 202.4, 202.5**) filter
on `"chainsync.transaction" / "chainsync.block" / "chainsync.rollback"`.

Adder renamed these in commit `f6d7094` (between `v0.37.0` and `v0.38.0`,
Feb 2026). The starter-kit pins `v0.39.1` and the library emits **`input.transaction`,
`input.block`, `input.rollback`**. `filter_event.WithTypes(...)` does an exact
`slices.Contains` match — no normalization — so the filter string
`"chainsync.transaction"` now matches zero events.

This is a bug **upstream in `adder-library-starter-kit`** too (the starter-kit
itself uses `"chainsync.transaction"` in `cmd/event-address-filter/main.go`).
Lessons are faithful to a broken starter-kit. Needs a coordinated fix.

### 🚨 Module 201 + 202 — Adder filter import path

Lessons import `github.com/blinklabs-io/adder/filter/chainsync`. That package
no longer exists — only `github.com/blinklabs-io/adder/filter/cardano` is
present in HEAD and in `v0.39.1`. The starter-kit uses the alias
`filter_chainsync` pointing at `filter/cardano`, which is the convention the
lessons should follow. The in-lesson "v0.36.0+ renames chainsync → cardano"
note is written as a *future* change; it's been reality for over a year.

Files:
- `202.3-store-and-retrieve-with-adder.md:57` — wrong import
- `202.4-select-query-provider.md:64` — wrong import; also cites a stale
  `~/go/pkg/mod/github.com/blinklabs-io/adder@v0.35.0/...` path twice

### 🚨 Module 203.2 — Apollo native-script API is fabricated

`203.2-mint-burn-native-script.md` uses an Apollo API that does not exist:

| Lesson says | Reality in Apollo HEAD |
|---|---|
| `apollob.AttachNativeScript(script)` | **No such method.** Apollo has `AttachV1Script / V2Script / V3Script` for Plutus only. Native scripts are added via the builder's internal `nativescripts` slice; there is no public setter. |
| `NativeScript.ScriptPubkey` | Typo — constant is `NativeScript.ScriptPubKey` (capital K). |
| `NativeScript.ScriptInvalidHereAfter` | Constant is `NativeScript.InvalidHereafter` (no `Script` prefix). |
| `NativeScript.NativeScript{Type: ..., KeyHash: ..., Slot: ...}` | Struct fields are `Tag, KeyHash, NativeScripts, NoK, Before, After`. No `Type` or `Slot` field. Use constructors: `NewScriptPubKey`, `NewInvalidHereafter`, `NewScriptAll`, etc. |
| `apollob.GetWallet().GetVerificationKey()` | The `Wallet` interface only exposes `GetAddress`, `SignTx`, `PkeyHash`, `SkeyHash`. No `GetVerificationKey`. Caller must type-assert to `*apollotypes.GenericWallet` and read `.VerificationKey`, or call `PkeyHash()` directly. |
| `vkey.PaymentKeyHash()` | `Key.VerificationKey` has `.Hash() (PubKeyHash, error)` — no `PaymentKeyHash` method. |
| `hex.EncodeToString(script.Hash())` | `script.Hash()` returns `(ScriptHash, error)` — two values. Won't compile as written. |
| `apollob.Submit()` on BlockFrost path | `Submit()` exists, but in 203.2 the ChainContext is `*BlockFrostChainContext`, so `bfc.SubmitTx(*tx)` is what the rest of the course uses. Mix is inconsistent with 102.2. |

Net: the minting code will not compile. Entire mint/burn flow needs rewriting
against the real API.

### 🚨 Module 203.1 — PlutusData struct fields are fabricated

`203.1-trace-aiken-type-to-go.md` constructs:
```go
PlutusData.PlutusData{
    TagNr:  121,
    HasTag: true,                               // field does not exist
    Value: PlutusData.PlutusIndefArray{
        PlutusData.PlutusData{
            HasTag: false,                      // field does not exist
            Value:  serialization.ByteString{Bytes: owner},  // type does not exist
        },
    },
}
```

Real Apollo `PlutusData` has three fields: `PlutusDataType (PlutusType), TagNr
(uint64), Value (any)`. There is **no `HasTag` field** and **no
`serialization.ByteString` type**. Apollo's own tests show the correct shape:

```go
PlutusData.PlutusData{
    TagNr:          121,
    PlutusDataType: PlutusData.PlutusBytes,
    Value:          []byte("Hello, World!"),
}
```

For a `Constr` with fields, wrap with `PlutusDataType: PlutusData.PlutusArray`
and `Value: PlutusData.PlutusIndefArray{...}`. The tag-index → CBOR-tag math
(`121 + index`) in the lesson is correct; only the Go struct shape is wrong.

### ⚠️ Module 201.3 — "WithPoolIds filters blocks" misleading

`201.3-filter-blockchain-events.md` Part 4 tells students to change the event
filter to `"chainsync.block"` and use `WithPoolIds`. That's also what the
starter-kit's dedicated `cmd/poolid-filter/main.go` does — good model — but
the lessons point at `cmd/event-address-filter` instead. Suggest pointing 201.6
at the dedicated command.

### ⚠️ Module 102.2 — inconsistent Apollo chain-context setup

The lesson creates a Blockfrost context **and** an empty backend:
```go
bfc, _ := BlockFrostChainContext.NewBlockfrostChainContext(...)
cc := apollo.NewEmptyBackend()
apollob := apollo.New(&cc)
```
then calls `bfc.Utxos(...)` and `bfc.SubmitTx(...)` directly, bypassing Apollo
entirely. This works but is confusing — a learner following the Apollo README
would simply pass `&bfc` to `apollo.New` and let the backend drive UTxO fetch
and submission. Suggest simplifying to match `readme.md:26-62` of Apollo.
